### ⚙️ 핵심 기능 구현을 위한 기술 키워드 및 상세 기능 정의

#### 1. 다중 포맷 자동 생성 (Multi-Format Content Generator)
*   **🎯 목표:** 하나의 원천 자산(Raw Video/Audio) $\rightarrow$ 여러 형태의 아웃풋 (블로그, 스레드, 영상 클립 등).
*   **기술 키워드:** `STT_Enhancement` (화자 분리 및 정확도), `Semantic_Chunking` (의미 단위 청킹), `Tone_Mapping`, `Prompt_Chaining`.
*   **구체적 기능 예시:**
    1.  **[영상 입력]** 30분짜리 강의 영상 파일을 업로드한다.
    2.  **[자동 처리]** AI가 이 영상을 분석하여 주요 주제(챕터)를 추출하고, 각 챕터별 하이라이트 구간을 자동으로 지정한다.
    3.  **[아웃풋 생성]**
        *   **블로그:** 챕터별 핵심 요약과 SEO 친화적인 제목/소제목 구조의 초안 생성.
        *   **X(구 트위터) 스레드:** 각 챕터당 5개의 개별적인 '훅(Hook)'이 담긴 카드뉴스 형식의 초안 자동 작성.
        *   **숏폼 클립:** 가장 임팩트 있는 발언 구간을 추출하여, 자막과 배경음악까지 입힌 60초 영상용 편집본 생성 (최소한의 수동 검토만 필요).

#### 2. 자동 회의록 및 지식 아카이브 시스템 (Knowledge Graph Builder)
*   **🎯 목표:** 비정형적인 음성/텍스트 정보를 구조화된, 검색 가능한 데이터로 변환한다.
*   **기술 키워드:** `STT_Speaker_Diarization` (화자 분리), `Entity_Recognition` (개체명 추출), `Action_Item_Extraction`, `Knowledge_Graphing`.
*   **구체적 기능 예시:**
    1.  **[녹음 파일 입력]** 클라이언트와의 30분짜리 미팅 녹취 파일을 업로드한다.
    2.  **[자동 처리]** AI가 대화 내용을 화자별로 분리하고(STT), 전문 용어/개체명(회사명, 제품명)을 인식하며 실시간으로 핵심 주제를 분류한다.
    3.  **[아웃풋 생성]**
        *   **구조화된 회의록:** (1) 논의 배경 $\rightarrow$ (2) 주요 안건별 토론 요약 $\rightarrow$ (3) **Action Item (담당자: OOO, 기한: YYY)**을 명확히 분리하여 표 형태로 제시.
        *   **지식 연결:** 미팅에서 언급된 모든 핵심 개념(예: '클라우드 의존성', 'STT 엔지니어링')을 개인의 지식 노트에 자동 링크하고 관련 자료를 추천한다.

#### 3. 톤앤매너 기반 배포 자동 최적화 (Tone-Aware Distribution Optimizer)
*   **🎯 목표:** 콘텐츠가 전달되는 채널과 청중에 따라 메시지의 '어조'와 '구조'를 자동으로 수정하여 최대의 도달률을 확보한다.
*   **기술 키워드:** `Sentiment_Analysis` (감성 분석), `Audience_Persona_Modeling`, `Contextual_Reframing` (문맥 재구성).
*   **구체적 기능 예시:**
    1.  **[원본 텍스트 입력]** 원본 콘텐츠(예: 깊이 있는 기술 블로그 포스팅)를 업로드한다.
    2.  **[배포 채널 지정]** 사용자가 '링크드인'과 '브런치' 두 채널을 선택하고, 타깃 청중을 '개발자 및 기업 의사결정자'로 설정한다.
    3.  **[자동 최적화]**

        *   **브런치 버전 (개인 브랜딩/교육):** 전문 용어에 대한 쉬운 설명을 추가하고, 질문과 답변 형식으로 구조를 재구성하여 독자의 몰입도를 높임.

## 🔍 P0-1: [Content Repurposing Engine] 상세 정의
**기능 목표:** 사용자가 생성한 하나의 긴 음성/영상 콘텐츠(강연, 인터뷰 등)를 다양한 온라인 채널에 맞춘 재활용 자산으로 자동 분리 및 가공한다.

### 🛠️ 최소 데이터 입력 (Minimum Input Data)
1.  **원천 미디어 파일:** 오디오 또는 비디오 파일 (MP3/WAV/MP4 권장).
2.  **메타데이터:** 콘텐츠의 주제, 목표 대상 청중(예: 초보 개발자), 원하는 출력 포맷 목록 (체크박스 선택).

### 💡 구체적인 최소 출력 정의 (Specific Output)
*   **1. 블로그/뉴스레터용 본문 초안 (Text):**
    *   출력 형태: Markdown 형식의 서술형 텍스트.
    *   요약 로직: 원본 파일에서 가장 핵심 주제가 언급된 구간을 자동 감지하고, 이를 구조화된 문단(서론-본론-결론)으로 재구성한다. (분량 제한: 500자 내외).
    *   **핵심 요소:** 제목 후보 3개와 각 포맷에 맞는 핵심 키워드 태그 목록.

*   **2. 카드뉴스/숏폼 대본 분할 (Text Snippets):**
    *   출력 형태: 리스트 형식의 짧은 문장(Snippet) JSON 배열.
    *   요약 로직: 원본 파일에서 '명령어적', '팁 제공형' 발언을 감지하여 10~20초 분량으로 나눌 수 있는 독립적인 메시지를 추출한다. (총 5개 이상).

## 📚 P0-2: [Voice-to-Structured Knowledge Base] 상세 정의
**기능 목표:** 사용자의 일상 회고, 학습 노트 등 음성 기반의 파편화된 지식 기록을 '검색 가능한 자산' 형태로 구조화하여 데이터베이스화한다.

### 🛠️ 최소 데이터 입력 (Minimum Input Data)
1.  **회고/학습 녹음 파일:** 오디오 파일 (MP3 권장). (예: "오늘 배운 것", "아이디어 메모")
2.  **필수 메타데이터:** 기록의 종류 지정 (예: '업무 회고', '개인 학습', '사업 아이디어').

### 💡 구체적인 최소 출력 정의 (Specific Output)
*   **1. 구조화된 지식 카드 (Knowledge Card):**
    *   출력 형태: Notion/Obsidian 등에서 사용 가능한 마크다운 포맷의 개별 노트.
    *   구조화 로직: STT를 통해 텍스트를 추출한 후, LLM이 해당 텍스트를 다음 세 가지 핵심 필드에 매핑한다.
        1.  **핵심 개념 (Concept):** 이 기록에서 가장 중요하게 다루어진 명사적 주제.
        2.  **관련 Pain Point:** 이 지식이 해결할 수 있는 문제점(키워드 태그).
        3.  **행동 지침 (Action Item):** 해당 지식을 바탕으로 당장 시도해볼 만한 구체적인 다음 행동 목록.

### ⚙️ 개발팀 전달용: 상세 요구사항 정의 방향 (Developer Handover)**

2.  **[분석 파이프라인]**: STT $\rightarrow$ NLP (요약/주제 분리) $\rightarrow$ 포맷팅 엔진 순으로 동작하며, 생성된 자산 목록(블로그 초안, 태그 등)을 사용자에게 제시해야 합니다.
3.  **[자원 관리 시스템]**: 모든 기능 사용은 **'크레딧'**을 소모하도록 설계합니다. (예: 1시간 분량 분석 = X 크레딧). 이 크레딧 차감 로직이 가장 중요합니다.

# 📑 크레딧 소비 트랜잭션 엔진 API 명세 (Credit Consumption Engine)

## 1. 목표 및 원칙
*   **목표:** 사용자가 특정 기능을 사용할 때, 요청된 기능의 비용을 계산하고, 사용자 계정에서 해당 크레딧을 **원자적으로(Atomically)** 차감하는 API를 제공한다.
*   **핵심 원칙 (Atomic Transaction):** 크레딧 검증 $\rightarrow$ 차감 성공/실패 판단 $\rightarrow$ 트랜잭션 완료는 반드시 순차적이고 실패 지점 처리가 명확해야 한다.

## 2. 엔드포인트 정의
| 항목 | 내용 | 상세 설명 |
| :--- | :--- | :--- |
| **HTTP Method** | `POST` | 자원 생성(소비) 행위를 나타내므로 POST 사용이 적절함. |
| **Endpoint URL** | `/api/v1/consume-credit` | 명확한 버전 관리를 위해 v1을 붙임. |
| **인증 (Auth)** | Bearer Token (JWT 권장) | 사용자 식별 및 접근 제어에 필수적. 토큰 발급 주체는 Auth Service가 담당한다고 가정함. |

## 3. 요청 페이로드 (Request Payload - JSON)
클라이언트(프론트엔드/API 게이트웨이)가 백엔드로 보내야 하는 최소한의 정보입니다.

```json
{
  "user_id": "uuid-generated-user-id", // 필수: 누가 소비하는지 식별 (billing 및 로그 추적용)
  "source_asset_type": "AUDIO",         // 필수: 원천 자산 타입 (예: AUDIO, VIDEO, TEXT). 로직 검증에 사용.
  "function_scope": "qa_pair_definition", // 필수: 현빈의 비용 테이블에 매핑되는 구체적인 기능 ID (enum 권장)
  "quantity": 3                        // 필수: 소비할 단위 개수 (예: Q&A 페어 3개 세트). 기본값은 1.
}
```

### ✅ `function_scope` 매핑 가이드라인 (Backend Enum/Config)
현빈이 정의한 비용 테이블을 기반으로, 백엔드에서 사용할 표준화된 ID 리스트입니다. 이 매핑 오류가 가장 흔하게 발생할 수 있으므로 Strict Validation이 필요합니다.

| 기능 설명 | `function_scope` (ID) | Cost per Unit | 비고 |
| :--- | :--- | :--- | :--- |
| 핵심 키워드 요약 | `keyword_summary` | 1 | 가장 기본 단위. |
| 소셜 미디어 콘텐츠 패키지 | `social_package` | 2 | 세트당 비용 계산. |
| Q&A 페어 정의 | `qa_pair_definition` | 3 | |
| 블로그 초안 생성 | `blog_draft_generation` | 5 | |
| 비디오 스크립트 아웃라인 생성 | `video_outline_creation` | 7 | |
| 지식 지도 구축 (KB Map) | `knowledge_base_map` | 8 | 가장 고가치 기능. |

## 4. 응답 구조 (Response Payload - JSON)

### A. 성공 시 응답 (HTTP Status: `200 OK`)
요청이 성공적으로 처리되었으며, 실제 차감된 크레딧과 남은 잔액을 명확히 반환합니다.

```json
{
  "success": true,
  "message": "Credit deduction successful.",
  "data": {
    "total_cost_deducted": 15, // 소비된 총 크레딧 (예: 3 * 5)
    "new_balance": 85,        // 차감 후 사용자 잔액
    "remaining_credits": true   // 잔여 크레딧이 남아있음을 bool로 표시하여 프론트엔드에서 UI 처리가 용이함.
  }
}
```

### B. 실패 시 응답 (HTTP Status: `4xx`)

#### 1. Insufficient Funds (가장 중요)
사용자 잔액 부족으로 인한 차단입니다.
*   **Status:** `403 Forbidden`
*   **Body:**
    ```json
    {
      "success": false,
      "code": "INSUFFICIENT_FUNDS",
      "message": "Required credits (15) exceed current balance (10). Please top up.",
      "required_cost": 15,
      "current_balance": 10
    }
    ```

#### 2. Invalid Scope/Parameter
요청된 기능이나 파라미터가 유효하지 않을 때 발생합니다.
*   **Status:** `400 Bad Request`
*   **Body:**
    ```json
    {
      "success": false,
      "code": "INVALID_PARAMETER",
      "message": "The provided function_scope 'xyz_undefined' is not recognized or requires additional parameters.",
      "details": {
        "function_scope": ["keyword_summary", "qa_pair_definition", ...] // 유효한 스코프 리스트 제공
      }
    }
    ```

## 5. 기술적 검토 및 추가 고려 사항 (Kodari's Checklist)
1.  **Rate Limiting:** 이 엔드포인트는 트래픽 부하가 크므로, IP/User ID 기반의 Rate Limiter를 반드시 적용해야 합니다. (예: 분당 최대 10회 요청).
2.  **비동기 처리:** 만약 `knowledge_base_map`처럼 시간이 오래 걸리는 무거운 작업이 이 API 호출을 트리거한다면? 크레딧 차감은 즉시 완료하되, 실제 '처리'는 별도의 워커(Worker Queue: RabbitMQ/Kafka)로 분리하고, 사용자에게 **Job ID**를 반환해야 합니다.
3.  **롤백 로직:** 트랜잭션 관리 시 데이터베이스 레벨의 롤백을 철저히 구현하여, 크레딧 차감과 내부 상태 기록(Usage Log)이 동시에 커밋되도록 해야 합니다.

#### 🟡 [Phase 2] 고통점(Pain Point) 심층 탐구: 시나리오 기반 질문 (15분)

**A. 콘텐츠 재가공 (Repurposing) Pain Point 탐색:**
1.  "어떤 큰 주제로 **하나의 핵심 자료(예: 1시간 강연, 긴 기사)**를 만들었다고 가정해 봅시다. 이 내용을 블로그 포스팅, 인스타그램 카드뉴스, 유튜브 쇼츠 등 여러 채널에 뿌려야 할 때, 가장 많은 시간을 소모하는 단계는 무엇인가요?" (→ 수작업의 비효율성 확인)
2.  "만약 콘텐츠 A가 너무 좋아서 리메이크(Re-make)해서 3개월 뒤 다시 쓰고 싶을 때, 그 원본 자료를 찾거나 참고하는 과정은 어떠신가요? 자료 보관/검색 시스템에 불편함이 있나요?" (→ 아카이브 및 연결성 확인)

**B. 음성 지식 베이스 추출 (Voice KB Extraction) Pain Point 탐색:**
1.  "최근 회의나 미팅을 녹음한 파일이나, 중요한 아이디어가 담긴 오디오 파일을 다루셨을 때가 있으신가요? 그 내용을 나중에 '자료'로 활용하기 위해 어떤 작업을 거치시나요?" (→ 수동 전사(Transcription) 및 요약 작업량 확인)
2.  "녹취된 파일에서 핵심 키워드나 액션 아이템만 쏙 뽑아 구조화된 지식 베이스(Knowledge Base)를 만들고 싶은데, 지금은 어떤 방식으로 하시나요? (직접 메모, 다른 사람에게 부탁 등)" (→ 자동 구조화의 필요성 확인)

*   **목표:** 우리의 기능(P0)을 간접적으로 제시하고 반응을 측정한다.
*   **질문 예시:**
    1.  "만약 녹음된 파일에서 AI가 핵심 요점과 액션 플랜만 자동으로 추출해서, 카카오톡으로 정리본이 오는 서비스가 있다면, 지금 가장 궁금한 정보는 무엇일까요?" (→ 기능의 가치 인지)
    2.  "(잠시 멈춘 후) 이런 과정을 '자동화'하는 것이 가능하다면, 이 시간 절약에 대해 기꺼이 투자할 만하다고 느끼실 수 있는 비용은 어느 정도라고 생각하시나요? (예: 월 구독료로 몇만 원대까지 괜찮다 등)" (→ WTP 측정)

## II. 사용자 플로우 및 KPI 정의 (Funnel Mapping)

MVP의 성공은 **무료 체험자가 얼마나 깊이 제품에 몰입하여, 자발적으로 결제 장벽까지 도달하는가**에 달려 있습니다. 아래 4단계 플로우는 사용자의 여정(Journey)과 그 단계별로 추적해야 할 KPI를 정의합니다.

| :--- | :--- | :--- | :--- | :--- |
| **Stage 1: 유입 및 체험** | 💡 **가입 및 온보딩:** 랜딩 페이지 $\rightarrow$ 무료 계정 생성. 초기 가이드 콘텐츠 소비. | **0 크레딧 (제공)** | **Sign-up to Feature View Rate:** 회원가입 후, 핵심 기능(P0)의 상세 화면을 본 비율. (최소 70%) | **Bounce Rate < 30%:** 랜딩 페이지 이탈률 관리. |

| **Stage 3: 병목 구간 도달** | 🔥 **한계 인식 및 재가공 요청:** "더 많은 콘텐츠를 만들고 싶은데, 크레딧이 부족하다." $\rightarrow$ 결제 페이지 진입을 고려함. | **크레딧 소진 경고 (Paywall Trigger)** | **Free-to-Paid Conversion Rate:** 무료 사용자가 유료 플랜의 가격 페이지에 도달한 비율. # 💰 WTP 기반 3대 가격 모델 시뮬레이션 및 비교 분석

## 핵심 전제 (Pain Point)
*   **사용자 고통점:** 녹음된 음성 파일이나 산발적인 메모(원천 데이터)가 구조화되지 않아, 재활용 가능한 자산으로 만들 수 없음.
*   **우리가 제공하는 가치:** 원천 데이터를 '블록 단위의 연결된 지식 자산'으로 변환하고, 이를 다양한 플랫폼에 통합할 수 있는 시스템.

## 📊 모델 A: 시간 절약 기반 가격 책정 (The Efficiency Pitch)
**핵심 논리:** "당신의 노동 시간을 아껴드립니다." 사용자가 콘텐츠를 직접 만들거나 구조화하는 데 투입했을 시간을 AI가 대신 줄여주는 것이 핵심 가치입니다.

| 구분 | 내용 | 설명 |
| :--- | :--- | :--- |
| **가격 메커니즘** | 고정 월 구독료 (프리미엄 티어) + 시간당 크레딧 패키지. | 사용자의 예상 업무량(시간)을 추정하여 단계별로 가격을 책정합니다. |
| **적용 시나리오** | "하루 2시간씩 블로그 초안 작성 및 요약 작업이 필요한 프리랜서 A에게 월 30만원의 효율화 패키지를 판매." | 시간 단위로 명확히 가치를 정의하여, 높은 초기 진입 장벽을 만듭니다. |
| **강점 (Strengths)** | 1. **WTP 인지 용이성:** '돈 = 시간'이라는 보편적 원칙에 기반해 설득력이 높습니다. 2. **프리미엄 포지셔닝:** 높은 월 구독료가 제품의 가치를 극대화합니다. |
| **약점 (Weaknesses)** | 1. **측정 난이도(MVP):** 사용자의 '시간 단축분'을 정확하게 측정하기 어렵고, 이는 KPI 설계에 복잡성을 더합니다. 2. **경쟁 심화:** 모든 SaaS가 시간 절약을 주장하므로 차별화가 어려울 수 있습니다. |

## 📚 모델 B: 자산 규모 기반 가격 책정 (The Asset Growth Pitch)
**핵심 논리:** "당신의 지식 체계(자산)를 구축하는 데 투자하세요." 사용자가 플랫폼에 축적하는 '지식의 양과 연결성' 자체에 가치를 부여합니다.

| 구분 | 내용 | 설명 |
| :--- | :--- | :--- |
| **가격 메커니즘** | 누적 자산 용량(예: 10GB), 통합 블록 개수, 또는 연결된 데이터베이스 관계 수에 따른 티어링 구독료. | 단순히 사용한 크레딧이 아니라, **사용자가 플랫폼에 '남기는 결과물'의 가치**를 기준으로 부과합니다. |
| **적용 시나리오** | "월 50GB까지 원천 데이터를 구조화하여 Notion DB로 연결하고 싶다면, 상위 티어(₩59,000)가 필요합니다." | 사용자가 플랫폼에 종속될수록 (자산을 쌓을수록) 더 높은 가격대의 구독을 하도록 유도합니다. |
| **강점 (Strengths)** | 1. **브랜드 철학 부합:** '원천 자산 기반 통합 콘텐츠 워크플로우'라는 당사의 핵심 가치를 가장 강력하게 반영합니다. 2. **Lock-in 효과 극대화:** 사용자가 많은 데이터를 쌓을수록 이탈 비용(Switching Cost)이 높아집니다. |
| **약점 (Weaknesses)** | 1. **초기 사용자 진입 장벽:** '용량'이나 '자산 규모' 같은 추상적 개념은 신규 사용자에게 이해하기 어려울 수 있습니다. 2. **데이터 측정 복잡성:** 자산의 가치를 정의하고 정확히 측정하는 백엔드 로직이 필요합니다. |

## 🗄️ 모델 C: 원천 데이터 볼륨 기반 가격 책정 (The Capacity Pitch)
**핵심 논리:** "처리할 재료(Raw Data)가 클수록, 더 많은 능력이 필요합니다." 투입되는 원본 데이터의 크기나 양에 비례하여 비용을 부과하는 방식입니다.

| **적용 시나리오** | "지난 분기 회의 녹취록 120시간을 한번에 구조화하고 싶습니다. 전문 분석 패키지(₩99,000)를 구매하세요." | 대규모 프로젝트 단위로 판매하기 때문에 초기 수익성이 높고 예측이 쉽습니다. |
| **강점 (Strengths)** | 1. **단순 명료성:** '몇 시간'이라는 직관적 단위가 있어 사용자가 이해하기 가장 쉽습니다. 2. **큰 규모의 매출 확보 용이:** 대규모 고객(에이전시, 소규모 팀)을 유치하는 데 유리합니다. |
| **약점 (Weaknesses)** | 1. **반복 구매율 저하 우려:** 초기 대용량 처리에 초점이 맞춰져 있어, 꾸준한 '운영 자동화'라는 지속적인 사용 동기를 부여하기 어렵습니다. 2. **가치 희석:** 단순 데이터 용량으로 인식되면, 우리의 핵심 가치(구조화/통합)가 퇴색될 수 있습니다. |

9.  $\downarrow$
10. **(Success)** $\rightarrow$ **[Credit Service]**: 최종 크레딧 차감 확정 (`COMMIT`) & 결과 반환
11. **(Failure/Timeout)** $\rightarrow$ **[Credit Service]**: 예약된 크레딧 롤백 (`ROLLBACK`)
