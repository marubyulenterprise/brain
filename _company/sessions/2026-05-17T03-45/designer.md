### 🛠️ Loss Funnel 컴포넌트: 에러 및 예외 상태 비주얼 자산 Spec Sheet (V3.0)

**목표:** 시스템의 모든 실패 지점(Failure Point)에서 사용자가 혼란을 느끼지 않고, 문제의 원인과 해결 방법을 명확하게 인지하도록 한다.
**기준:** 엔지니어링적 정밀함, 데이터 기반 메시징, 과장 없음.

#### 1. 에러 상태 유형 정의 및 우선순위

| 상태 ID | 사용자 상황 | 발생 주체 | 비주얼 중요도 | 기본 색상 코드 | 코멘트 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **ERROR\_API\_FAIL** | API 통신 실패, 네트워크 오류. (가장 흔함) | 백엔드/네트워크 | High | `#ffc107` (Amber/Warning) | 시스템 문제로 인한 경고. 재시도(Retry) 유도 필수. |
| **ERROR\_CREDIT** | 크레딧 부족, 결제 한도 초과 등 비즈니스 제한. | 인증/결제 로직 | High | `#6c757d` (Grey/Informative) | 기술적 실패가 아닌 '자원'의 문제임을 명확히 분리해야 함. |

| **STATE\_EMPTY** | 입력 데이터가 아예 없음. (No Data Input) | 프론트엔드 로직 | Low | `#adb5bd` (Light Grey) | 기능 자체가 작동하지 않는 것이 아니라, *데이터*가 부족함을 알려야 함. |

#### 2. 상태별 UI/UX 컴포넌트 및 애니메이션 가이드라인

##### A. 💾 `ERROR_API_FAIL` (기술적 실패: 데이터 연결 오류)
1.  **헤딩 메시지:** "데이터 로드에 실패했습니다." 또는 "시스템과 데이터를 동기화할 수 없습니다."
2.  **서브 메시지:** *원인 명시* (예: 네트워크 연결 상태를 확인하거나, 잠시 후 다시 시도해주세요.)
3.  **핵심 컴포넌트:** `RetryButton` (Primary CTA)
    *   **디자인:** 일반적인 버튼보다 약간 비활성화된 느낌의 외곽선만 있는 디자인을 사용한다.
    *   **애니메이션:** 버튼 클릭 시, 로딩 스피너가 50ms 간격으로 깜빡이며 재시도 로직이 실행됨을 보여준다. (기술적 노력 표현)
4.  **비주얼 자산:** 와이어프레임 형태의 깨진 데이터 패킷(Broken Data Packet) 아이콘 사용.

##### B. 💳 `ERROR_CREDIT` (경고성 실패: 크레딧 부족)
1.  **헤딩 메시지:** "활용 가능한 크레딧이 부족합니다." (가장 중요!)
2.  **서브 메시지:** *해결책 제시* (예: [추가 크레딧 구매] 또는 [요금제 업그레이드]를 통해 기능을 이용할 수 있습니다.)
3.  **핵심 컴포넌트:** `PurchaseCTA` (Secondary CTA)
    *   **디자인:** Loss Funnel의 최종 성공 CTA와 동일한 톤앤매너를 유지하되, 색상은 경고가 아닌 '행동 유도' 느낌을 강조한다.
    *   **애니메이션:** "크레딧 부족" 메시지가 나타날 때, 돈이나 에너지가 빠르게 줄어드는 듯한 `Pulse` 애니메이션 효과를 주어 시각적 긴급성을 부여한다. (재무적 위협 연계)

##### C. 🛠️ `ERROR_MAINTENANCE` (시스템 운영 중단: 점검 안내)
1.  **헤딩 메시지:** "서비스가 일시적으로 중단되었습니다."
2.  **서브 메시지:** *예상 복구 시간 명시* (예: 시스템 개선을 위해 금일 오후 3시부터 5시간 동안 점검이 예정되어 있습니다.)
3.  **핵심 컴포넌트:** `InfoNotice` (CTA 없음)
    *   **디자인:** 가장 차분한 배경색(옅은 회색)과 간결하고 안정적인 타이포그래피를 사용한다. 복잡한 애니메이션을 배제하여 신뢰감을 높인다.
4.  **비주얼 자산:** 톱니바퀴와 체크 표시가 함께 돌아가는 아이콘 (Maintenance Icon).
