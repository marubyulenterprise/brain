## ⚙️ PVU API v1.0: 데이터 정의 및 흐름도 (Data Dictionary & Lineage)

### 1. 시스템 목표 및 출력 값 재확인
| 항목 | 내용 | 비고 |
| :--- | :--- | :--- |
| **API 엔드포인트** | `POST /api/calculate-pvu` | PVU 계산 요청 수신 |
| **목표 지표 (Output)** | 잠재 추가 순이익 (Predicted Value Unit, PVU) | $$PVU = \frac{(A + B)}{C} \times Weight$$ |
| **핵심 비즈니스 정의** | PVU는 사용자가 현재 놓치고 있는 '잠재적 가치'를 수치화하여 제시함으로써 결제 유도를 하는 것이 목적이다. | 손실 방지(Loss Prevention) 관점의 설계가 필요하다. |

### 2. 입력 변수 상세 정의 (Data Schema & Validation Rules)

각 변수는 이름, 데이터 타입, 설명, 그리고 반드시 지켜야 할 **비즈니스 제약 조건**을 가집니다.

| 변수명 | 변수 역할 | 데이터 타입 | 필수 여부 | 비즈니스 제약/유효성 규칙 (Validation Rule) |
| :--- | :--- | :--- | :--- | :--- |
| **`asset_value` (A)** | 연결 자산 가치: 현재 확보된 원천 콘텐츠 및 수익화 가능성을 정량화. | `Float` | 필수 | `0 < A <= 1,000,000`. 단순히 '개수'가 아닌, *최소 추정 수익* 기반이어야 함. (예: $50/월) |
| **`opportunity_coeff` (B)** | 시장 기회 계수: 해당 콘텐츠에서 발견되는 외부 트래픽 또는 키워드 잠재력. | `Float` | 필수 | `0 < B <= 100`. 검색 볼륨(Search Volume) 대비 *현재 커버리지 부족 정도*를 반영해야 함. (최대치를 넘기지 않도록 제한 필요) |
| **`difficulty_cost` (C)** | 난이도/실행 비용: PVU를 실현하기 위해 필요한 시간, 노동력 및 개발 투입 비용. | `Float` | 필수 | `0 < C <= 5`. 너무 낮으면 가치가 없어지고, 너무 높으면 실행 동기가 상실되므로 합리적인 범위가 필요함. (1~5점 스케일 권장) |
| **`weighting_factor` (W)** | 시간 민감도 및 긴급성: 이 PVU를 해결하지 않았을 때의 시급성을 반영하는 가중치. | `Float` | 선택 | `0.5 <= W <= 2.0`. 일반적으로는 1.0으로 설정하되, 'Critical Alert' 발생 시만 높임(예: 2.0). |

### 3. 데이터 출처 및 흐름도 (Source of Truth & Data Lineage)

각 변수가 어떤 시스템에서 오는지 명확히 정의하여, 개발자가 API 연동 순서를 잡도록 합니다. **가장 먼저 필요한 통합은 A와 B입니다.**

| 변수 | 비즈니스 의미 | Source of Truth (데이터 원천) | 데이터 수집 주체/담당 에이전트 | 요구되는 API 엔드포인트 |
| :--- | :--- | :--- | :--- | :--- |
| **A** (`asset_value`) | 현재 보유 자산의 경제적 가치. | 1. 내부 DB (사용자 계정 정보) <br> 2. 결제 로그 시스템 (Stripe, PayPal 등) | Backend / Billing Agent | `/api/user/revenue-summary` |
| **B** (`opportunity_coeff`) | 시장이 원하는 잠재적 검색어 트래픽 규모. | Google Keyword Planner API <br> 전문 키워드 리서치 도구 데이터베이스 (내부 구축) | Researcher / Content Agent | `[API 연동 필요]` (Keyword Data Source) |
| **C** (`difficulty_cost`) | 해결책을 구현하는 난이도. | 사용자 입력 또는 내부 프로젝트 관리 시스템 (JIRA/Trello 등) 기반의 점수화된 데이터. | User / Project Manager Agent | `/api/project/effort-score` |
| **W** (`weighting_factor`) | 예측 경고 시점의 위기감. | AI 분석 엔진의 '임계치 감지 로직' (Critical Path Detection) | Business Strategy Layer (현빈 담당) | 내부 비즈니스 로직 판단 |

### 4. 개발팀 전달 요약 및 액션 아이템

**[개발자 코다리에게]**
1.  API 요청 시, A, B, C가 모두 포함되어야 하며, 누락된 변수에는 `MissingInputError`를 반환해야 합니다.
2.  A와 B는 단순한 입력값이 아니라 **외부 시스템과의 연동(Integration)**을 통해 채워지는 '흐름 데이터'라는 점을 인지해 주십시오. 이는 MVP 단계에서 가장 큰 개발 리소스가 될 것입니다.

**[다음 전략적 액션]**
데이터 출처가 명확해졌으므로, 다음 단계는 이 데이터를 실제로 수집하여 **PVU를 계산할 만한 최소한의 '테스트 데이터 세트'**를 확보하는 것입니다. B 변수(시장 기회 계수)의 데이터 연동이 가장 복잡하고 중요합니다.
