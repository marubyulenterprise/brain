# 📑 Payment Validation Layer (PVL) 비즈니스/UX 로직 명세서
**대상:** Developer Agent
**목표:** 사용자가 유료 결제 과정을 거쳐 $L_{Occ}$ 보고서를 다운로드하고, 실패 시에도 시스템적 신뢰도를 유지하며 재시도하도록 유도한다.

## 1. 성공 플로우 (Success Flow: Report Access Granting)

### A. 핵심 로직 정의
| 요소 | 상세 내용 | 비고 |
| :--- | :--- | :--- |
| **조건** | 결제 승인 상태(Payment Status: `SUCCESS`)이며, 최종 청구 금액이 특정 임계값($T_{min}$) 이상일 경우. | $T_{min}$은 최소 구매가 및 보고서 접근 권한에 해당함. (예시: ₩9,900) |
| **액션** | 1. 시스템 내부 기록 저장: 사용자 계정에 `Report_Access_Granted` 플래그 설정. <br>2. 프론트엔드 UX 트리거: 결제 완료 페이지에서 '보고서 다운로드' CTA 활성화 및 즉시 접근 권한 부여. | 이 플래그는 추후 이용 약관 위반 등의 상황에 대비해 DB 레벨에서 관리되어야 함. |
| **다운로드 경험** | 사용자에게 단순 PDF 파일 제공이 아닌, **[L_Occ 분석 보고서 - [날짜]]** 형태의 고유 ID가 박힌 전문적인 템플릿을 제공한다. (브랜드 신뢰도 확보) | 다운로드 성공 시, "분석 결과를 확인하셨습니다. 추가적인 시스템 진단을 원하시면..."과 같은 **다음 단계 CTA를 삽입**하여 재구매/업그레이드를 유도해야 함. |

### B. 가격 전략 연동
*   **권장 로직:** $L_{Occ}$ 보고서 다운로드 권한은 '분석 크레딧 패키지' 구매 시에만 부여되어야 합니다. (단발성 결제 모델 + 구독 전환 유도)
*   **결과물 명시:** "최소 ₩9,900 결제로 $L_{Occ}$ 보고서 다운로드 권한을 확보했습니다."

## 2. 실패/취소 플로우 (Failure/Cancel Flow: Error Handling & Recovery)

실패 상황에서는 **'시스템적 오류'**로 포지셔닝하여 사용자가 개인의 실수라고 느끼지 않게 해야 합니다. 기술적인 원인을 제시하고, 해결책을 명확히 알려주는 것이 핵심입니다.

### A. 상태별 메시지 및 로직 정의
| 실패 유형 | 시스템 코드 (사용자에게 노출) | 사용자 가이드 메시지 (톤앤매너: 전문적/기술적) | 재시도 유도 로직 |
| :--- | :--- | :--- | :--- |
| **결제 거절** (카드 만료, 한도 초과 등) | `[ERR-401]: Payment Validation Failed` | "지불 수단 정보가 현재 거래 기준을 충족하지 못했습니다. 카드사 정책 또는 잔액을 확인해 주세요." | [Retry with new card] 버튼 활성화 및 결제 수단 변경 유도. |
| **API 통신 실패** (네트워크 문제, 서버 다운) | `[ERR-503]: Service Unavailable` | "결제 시스템과 일시적인 연결 오류가 발생했습니다. 잠시 후 다시 시도해 주십시오." | 5분/10분 타이머를 보여주며 [Retry Later] 버튼 활성화. (자동 재시도 로직 포함) |
| **취소/이탈** (사용자가 중간에 나가버림) | N/A | "결제 과정이 완료되지 않았습니다. 필요한 크레딧을 다시 확인하고 결제를 시도해 주세요." | 실패 페이지가 아닌, 메인 Dashboard로 리다이렉트하며 '계속 진행' CTA를 노출하여 이탈 방지. |

### B. UX/UI 요구사항 (개발 참고)
1.  **오류 메시지 영역:** 일반적인 빨간색 경고창 대신, 시스템 모달(System Modal) 형태로 구현하여 권위를 높인다.
2.  **원인 분석 제공:** 오류가 발생했을 때, "이 문제를 해결하기 위해 확인해야 할 3가지 사항" (예: 카드 유효기간, 한도 체크, 네트워크 연결)을 리스트로 제공한다.
