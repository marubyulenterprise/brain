# 🛠️ Onboarding Flow: PVL 통합 Mockup 세트 (Final Version)
**대상:** Front-End / UI/UX 구현
**목표:** 결제 과정 전체를 '시스템의 권위적인 진단 및 승인' 경험으로 설계하여 이탈률을 낮추고 업그레이드 전환율을 높인다.

## 1. Paywall Mockup: [Access Restricted - $L_{Occ}$ 보고서]
**상황:** 사용자가 핵심 분석 데이터를 발견했으나, 최종 보고서(Loss Amount Report) 다운로드가 불가능할 때 나타난다.
**UX 목표:** '이 기능은 매우 귀중하며, 시스템이 이를 관리하고 있다'는 느낌을 주어 결제 행위를 필수적인 과정으로 인식시킨다.

| 요소 | 디자인 스펙 및 로직 | 기술적 근거/톤앤매너 |
| :--- | :--- | :--- |
| **헤드라인** | `[SYSTEM ALERT] $L_{Occ}$ 보고서 다운로드가 제한되었습니다.` | 크고, 경고성 폰트(Monospace) 사용. 시스템 메시지처럼 보이게 함. |
| **본문 (Diagnosis)** | "이 '손실 분석 보고서'는 단순히 데이터를 모은 PDF가 아닙니다. Maru Company의 독자적인 Payment Validation Layer를 통과해야만 생성되는, 고유 ID와 진단 시점이 기록된 전문 아티팩트입니다." | 기술적 용어(Validation Layer, Artifact) 사용으로 권위 확보. 일반적인 '결제' 언급 지양. |
| **시각적 강조** | 배경에 은은한 그리드 패턴을 넣고, 다운로드 버튼 주변에 미세하게 깜빡이는 (Pulse) 효과를 적용한다. | 시스템 작동 중임을 시각적으로 암시하여 신뢰도 높임. |
| **CTA 디자인 (핵심)** | **[🔑 보고서 접근 권한 획득하기]** (Primary Button, Gold Color) <br> *하단 작은 글씨:* 최소 크레딧 패키지(₩9,900) 구매로 즉시 접근 가능합니다. | CTA가 곧 '시스템의 잠금 해제'를 의미하도록 디자인. 가격을 마지막 순간에 제시하여 가치를 최대화함. |

## 2. Success Mockup: [SYSTEM ALERT - Access Granted]
**상황:** 사용자가 결제를 완료하고, 시스템이 보고서 접근 권한을 성공적으로 부여했을 때 나타난다.
**UX 목표:** 단순 거래가 아닌, '시스템으로부터 인증(Certification)'을 받았다는 느낌을 주어 만족도와 신뢰를 극대화한다.

| 요소 | 디자인 스펙 및 로직 | 기술적 근거/톤앤매너 |
| :--- | :--- | :--- |
| **헤드라인** | `✅ [SUCCESS] 시스템 승인 완료: $L_{Occ}$ 보고서 접근 권한 획득.` | 녹색 체크와 함께 'SYSTEM' 태그를 붙여 공식적인 느낌을 준다. |
| **승인 정보 박스 (핵심)** | **[Certificate of Access]** <br> *발급 ID:* `MARU-LXX-[YYYYMMDD]-[UNIQUEHASH]` <br> *발급 시각:* YYYY-MM-DD HH:mm:ss KST <br> *권한 만료일:* (만료 기간 명시) | **가장 중요.** 고유 식별자(ID, Hash)를 강제로 노출하여 물리적/디지털 자산의 가치를 극대화한다. 이 정보는 모바일 배경화면이나 출력물에 사용될 수 있다. |
| **다운로드 경험** | "보고서를 다운로드하거나 대시보드에서 즉시 열람하세요." 버튼 활성화. (Primary CTA) | 성공 페이지의 주 목표를 '사용자가 다음 행동(다운로드)을 하도록 유도'하는 것이다. |
| **후속 액션 유도** | *추가적인 시스템 진단이 필요합니다.* <br> "현재 분석된 $L_{Occ}$는 부분적입니다. 다른 자원과의 연결성을 검토하여 [Missing Edge]를 발견하고, 추가 보고서를 확보하세요." (Secondary CTA) | 성공 직후 곧바로 '다음 문제'와 '업그레이드 기회'를 던져 재구매/구독을 유도한다. |

## 3. Failure Mockup: [SYSTEM ERROR - Action Required]
**상황:** 결제 과정 중 오류가 발생했거나, 시스템이 임계치에 도달하여 접근이 거부되었을 때 나타난다.
**UX 목표:** 사용자에게 책임을 돌리지 않고, '시스템적 문제'로 포지셔닝하며 해결 방법을 제시한다.

| 요소 | 디자인 스펙 및 로직 | 기술적 근거/톤앤매너 |
| :--- | :--- | :--- |
| **헤드라인** | `🚨 [SYSTEM ERROR CODE 503]: Payment Validation Failed.` | 빨간색 경고창(Alert Box)로 전체 화면을 가리고, 코드를 명시하여 기술적 권위를 유지한다. |
| **오류 진단 메시지** | "결제 게이트웨이와의 통신에 일시적인 오류가 발생했습니다. 이는 사용자 측의 문제가 아닌, 시스템 간의 데이터 전송 과정에서 발생한 비정상 상황입니다." | 명확히 '사용자 책임 아님'을 선언하여 불안감을 해소한다. |
| **원인 분석 (진단)** | **[원인 진단]** <br> *가능성 A:* 카드사 임시 승인 실패 (가장 흔함) <br> *가능성 B:* 게이트웨이 트랜잭션 시간 초과 <br> *해결책:* 5분 후 재시도 또는 다른 결제 수단을 이용해주세요. | 추상적인 '오류' 대신, 엔지니어가 분석하는 듯한 구체적이고 다단계의 진단 과정을 보여준다. |
| **CTA 디자인 (핵심)** | 1. **[🔁 재시도 (Retry)]** (Primary Button): 실패 코드를 자동으로 초기화하고 버튼을 활성화한다. <br> 2. **[ℹ️ 결제 가이드 확인]** (Secondary Link): FAQ/결제 방법 안내 페이지로 이동시킨다. | 사용자가 즉각적으로 할 수 있는 액션(Retry)을 최우선으로 배치한다. 실패가 끝이 아님을 인지시킨다. |
