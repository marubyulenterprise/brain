### 🛠️ [High Priority] 마이크로서비스 API 백로그 정의

#### 🎯 선정 API Endpoint
`POST /api/create-knowledge-graph`

#### 💡 기술적 근거 (Why?)
1.  **핵심 가치 구현:** 이 엔드포인트는 단순 요약(Journey 1)을 넘어, 다중 입력 간의 복잡한 관계(A $\to$ B, A $\leftrightarrow$ C 등)를 추출하고 그래프 DB에 원자적 트랜잭션으로 기록하는 역할을 합니다. 이것이 곧 '지식 자산화' 경험 그 자체입니다.
2.  **비용 구조 반영:** 현빈님이 정의한 `Pro Tier`의 핵심 가치(Deep Insight)와 직접적으로 연결됩니다. 사용자가 가장 큰 고통점(Pain Point)을 해결할 수 있는 지점이므로, 여기에서 크레딧 소모가 발생하고 가치가 증명되어야 합니다.
3.  **아키텍처 통합:** 이 API의 성공적인 구현은 기존에 정의된 '크레딧 검증 엔진' 및 '원천 자산 기반 지식 레이어' 전체 아키텍처를 강제로 한 번에 테스트하게 만듭니다.

#### ⚙️ 개발 상세 범위 (Scope Definition)
| 항목 | 내용 | 비고 |
| :--- | :--- | :--- |
| **목표** | 여러 개의 독립적인 지식 노드(Node A, Node B...)를 입력받아, 이들 간의 논리적/시간적 관계(Edge)를 추출하고, 이를 Graph DB에 트랜잭션으로 기록 및 반환. | 최종 결과물은 단순 텍스트가 아닌, '구조화된 그래프 데이터'여야 함. |
| **입력 파라미터** | `files`: 다중 파일 업로드 (PDF, TXT, Markdown 등).<br>`user_context`: 사용자가 추가적으로 제공하는 컨텍스트(예: "이 프로젝트의 목표는 X입니다").<br>`transaction_id`: 크레딧 트랜잭션 관리를 위한 고유 ID. | 입력 데이터가 비정형적이므로 유효성 검증 로직 강화 필요. |
| **핵심 처리 로직** | 1. 파일 단위 임베딩 및 노드(Node) 추출.<br>2. 상호 간의 Semantic Similarity 계산 및 관계 추출 (Edge).<br>3. `Credit Gateway`를 통해 트랜잭션 크레딧 검증 및 차감 (원자적 필수).<br>4. Graph DB에 Node/Edge 쌍으로 저장. | 이 과정에서 실패 시 롤백(Rollback) 로직이 가장 중요함. |
| **출력 응답 구조** | `{ success: boolean, message: string, graph_data: { nodes: [Node], edges: [Edge] }, remaining_credits: number }` | 성공 여부와 함께 실제 그래프 구조를 반환해야 클라이언트가 다음 액션(CTA)을 설계할 수 있음. |

#### 🚧 기술적 난이도 및 우선순위
*   **난이도:** 🔥🔥🔥🔥 (Very High)
    *   이유: 단순 백엔드 구현 문제가 아니라, NLP/AI 모델링 + 분산 트랜잭션 관리(ACID 원칙 준수)가 결합된 고도의 시스템 설계 문제이기 때문. 특히 크레딧 소모 로직과 데이터 영속성 사이의 **원자적 묶음 처리**가 핵심입니다.
