## 🎯 목적
본 문서는 Maru Company의 모든 핵심 서비스 로직(API Gateway를 경유하는 트랜잭션)에서 발생할 수 있는 **데이터 결핍(Data Deficit)** 및 **시스템 제약 실패** 시, 사용자에게 재무적 손실액($L$ Amount)을 산출하고 고지하기 위한 표준 계약서 역할을 합니다.

개발자는 이 기준표에 명시된 원인 분류 체계와 계산 로직을 따라 트랜잭션의 성공/실패 여부를 판단해야 합니다.

## 🔍 $L$ Amount 개념 정의 및 기본 공식

**1. 손실액($L$ Amount)이란?**
사용자가 현재 가지고 있는 **원천 자산(Source Asset)**이 부족하거나, 시스템 제약으로 인해 필요한 핵심 가치 변환을 완료하지 못함으로써 발생한 '잠재적 기회비용'이자 '미래 매출 규모의 추정치'입니다.

**2. 기본 산출 공식:**
$$L \text{ Amount} = (\text{원천 자산 부족량} \times \text{핵심 가치 변환 단가}) + \text{최소 보상료}$$

*   **원천 자산 부족량:** 필요한 데이터의 절대적 양 (예: 분량, 이미지 해상도, 길이).
*   **핵심 가치 변환 단가:** 해당 원천 자산을 우리 서비스만의 독점 로직(STT 전처리 $\rightarrow$ 지식 구조화 $\rightarrow$ 콘텐츠 재활용)을 거쳐 상품화할 때 발생하는 평균 수익성 (₩/단위).
*   **최소 보상료:** 트랜잭션 실패를 인지한 사용자에게 심리적 마찰과 불편함에 대해 최소한으로 제공하는 금액 (일반적으로 ₩5,000~₩10,000 사이의 임계값 설정 필요).

## 📋 Failure Scenario 및 $L$ Amount 산출 기준표

### I. 원천 데이터 품질(Source Data Quality) 기반 결핍 시나리오
| 오류 코드 | 실패 유형 (Pain Point) | 발생 조건/원인 분류 체계 | $L$ Amount 계산 로직 상세 | 예시 ($L$ Amount) |
| :---: | :--- | :--- | :--- | :--- |
| `ERR_DATA_RES` | **저해상도 이미지** (Low Resolution Image) | 원천 이미지가 AI 분석에 필요한 최소 DPI(예: 300dpi) 미만일 경우. OCR 정확도가 급격히 저하됨. | $L = (\text{필요 DPI} - \text{실제 DPI}) \times 1,500\text{원}$ (단위당 손실액). 최소 보상료 포함. | 이미지 해상도 부족으로 인한 지식 추출 불가 예측 매출: **₩30,000** |
| `ERR_DATA_VOICE` | **음성 데이터 부족/불명확** (Ambiguous Audio) | 음원 길이가 너무 짧거나, 배경 소음(Noise Floor)이 높아 STT 정확도가 임계점(예: 85% WER) 미만일 경우. | $L = (\text{최대 처리 가능 분량} - \text{실제 분량}) \times 12,000\text{원}/\text{시간}$. (STT의 가치 반영). | 음성 데이터 부족으로 인한 콘텐츠 기획 불가 예측 매출: **₩60,000** |
| `ERR_DATA_SOURCE` | **원본 자료 소유권/접근 제한** | 사용자가 핵심 데이터를 업로드했으나, AI가 구조화하는 과정에서 '맥락적 단절'이 발생하여 재활용 가능한 원천 정보 자체가 부족할 경우. | $L = (\text{총 데이터 볼륨} - \text{연결된 세그먼트 볼륨}) \times 5,000\text{원}/\text{세그먼트}$. | 자료 간 연결고리(Context) 부재로 인한 자동화 실패 예측 매출: **₩25,000** |

### II. 시스템 처리 로직/제약 기반 결핍 시나리오
| 오류 코드 | 실패 유형 (Pain Point) | 발생 조건/원인 분류 체계 | $L$ Amount 계산 로직 상세 | 예시 ($L$ Amount) |
| :---: | :--- | :--- | :--- | :--- |
| `ERR_LIMIT_API` | **처리 용량 초과** (Rate Limit Exceeded) | 단일 트랜잭션에서 처리할 수 있는 최대 데이터 크기(예: 20GB)를 초과하는 경우. | $L = (\text{요청된 용량} - \text{최대 허용 용량}) \times 5,000\text{원}/\text{GB}$. (시스템 확장 필요성 강조). | 처리 한계로 인한 대규모 데이터 분석 실패 예측 매출: **₩150,000** |
| `ERR_PROCESS_LOGIC` | **논리적 구조화 불가능** (Schema Mismatch) | 입력된 원천 데이터가 시스템이 정의한 핵심 스키마(예: 비즈니스 목표 $\rightarrow$ 액션 아이템)를 따르지 않아, 자동화된 지식 베이스 변환에 실패하는 경우. | $L = (\text{기대되는 구조적 출력 수} - \text{실제 추출 가능 항목 수}) \times 2,000\text{원}/\text{개}$. (가장 높은 가치 손실). | 구조화 패턴 미준수로 인한 핵심 아이디어 도출 실패 예측 매출: **₩45,000** |

### III. 사용자 인터페이스/경계 조건 기반 결핍 시나리오
| 오류 코드 | 실패 유형 (Pain Point) | 발생 조건/원인 분류 체계 | $L$ Amount 계산 로직 상세 | 예시 ($L$ Amount) |
| :---: | :--- | :--- | :--- | :--- |
| `ERR_AUTH` | **권한 부족** (Permission Denied) | 사용자가 결제 또는 핵심 기능 접근에 필요한 인증(OAuth/결제) 절차를 완료하지 않은 경우. | $L = \text{최소 보상료} + (\text{Pro Plan 전환 유도 금액})$. (직접적 매출 연결). | Pro 기능을 사용하려면 결제가 필요하다는 명확한 재무 고지: **₩10,000** |
| `ERR_INPUT_INVALID` | **입력값 형식 오류** (Format Mismatch) | API 요청 시 필수 필드(Required Field)가 누락되거나, 데이터 형식이 잘못된 경우. | $L = \text{최소 보상료} + (\text{재시도 횟수 제한 페널티})$. (낮은 손실액으로 사용자 교육에 활용). | 단순 입력 오류로 인한 트랜잭션 실패: **₩5,000** |

---
***[개발자 유의 사항]***
1. 모든 $L$ Amount는 **절대적인 금전적 피해액이 아닌, 잠재적 기회 상실 가치(Potential Loss)**임을 사용자에게 명확히 고지해야 합니다.
2. 트랜잭션 실패 시, 에러 메시지는 반드시 `[오류 코드]: [실패 유형] - (원인)으로 인해 예상되는 손실액은 ₩XXX입니다.` 형태로 구성되어야 합니다.
