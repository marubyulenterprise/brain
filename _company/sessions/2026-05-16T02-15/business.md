## 🎯 MVP v1.0 핵심 여정 정의: 관계성 추출 (Relationship Mapping)

**핵심 목표:** 사용자가 업로드한 단일 원본 문서 $\rightarrow$ 시스템이 엔티티(Node)를 식별하고, 이들 간의 상호작용(Edge)을 자동 매핑하여 구조화된 지식 그래프 생성.
**가치 증명 포인트 (WTP):** 단순히 텍스트를 요약하는 것이 아니라, '정보에 숨겨진 연결고리'와 '활용 가능한 자산 구조'를 제공함.

### I. 단계별 요구사항 정의 및 플로우 시퀀스

| 단계 | 주체 | 활동/액션 | 입력 데이터 (Input) | 출력 결과물 (Output) |
| :--- | :--- | :--- | :--- | :--- |
| **1. 진입** | 사용자 | 원본 콘텐츠 업로드 | 텍스트 파일 (.txt, .md) 또는 복사 붙여넣기된 긴 텍스트 블록 (최대 5,000자 권장) | - |
| **2. 처리 요청** | 시스템 | '지식 그래프 빌드' 실행 버튼 클릭 | 원본 콘텐츠 + 사용자가 설정한 원하는 노드/엣지 유형 필터 (선택적) | 크레딧 차감 시작 (예: 5 크레딧 소모) |
| **3. 추출 및 구조화** | 시스템 (AI Core) | ① 엔티티(Node) 식별, ② 관계(Edge) 정의 수행 | 원본 콘텐츠 텍스트 전체 | Node 목록, Edge 목록 (JSON 또는 내부 DB 트랜잭션 데이터) |
| **4. 결과 제시** | 시스템 | 사용자에게 시각화된 그래프 제공 및 상세 내역 보고서 출력 | ① 구조화된 노드/엣지 데이터 ② 원본 콘텐츠의 핵심 요약 | 1. 인터랙티브 지식 그래프 (시각적 증거). 2. 정의된 Node & Edge 리스트와 설명. |

### II. 최소 필수 데이터 유형 및 스키마 설계

가장 낮은 비용으로 MVP를 구현하기 위해, 초기에는 노드와 엣지의 타입을 **극도로 제한**하여 시작해야 합니다.

#### 1. 노드 (Node) 정의
| 필드명 | 데이터 타입 | 설명 | 예시 값 | 필수 여부 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `node_id` | UUID/Int | 고유 식별자 | `uuid-12345` | 필수 | 내부 관리용 |
| `type` | String (Enum) | 노드의 분류 체계 (핵심 가치) | **[Concept]**, **[Person]**, **[Tool]**, **[Topic]** | 필수 | 초기 4가지 유형으로 제한 |
| `name` | String | 추출된 핵심 엔티티 명칭 | '지식 그래프', '파이썬 언어', '마루 회사' | 필수 | |
| `description` | Text | 해당 노드에 대한 시스템 정의/요약 (출처 기반) | "엔지니어가 사용하는 데이터 구조화 방법론." | 선택 | 지불 가치 상승 요인 |

#### 2. 엣지 (Edge) 정의
| 필드명 | 데이터 타입 | 설명 | 예시 값 | 필수 여부 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `edge_id` | UUID/Int | 고유 식별자 | `uuid-67890` | 필수 | 내부 관리용 |
| `source_id` | Node ID | 관계의 시작점 (Source Node) | `node-12345` | 필수 | 노드 A $\rightarrow$ 노드 B |
| `target_id` | Node ID | 관계의 끝점 (Target Node) | `node-67890` | 필수 | |
| `type` | String (Enum) | 관계의 성격 (핵심 가치) | **[Influences]**, **[RelatesTo]**, **[IsUsedBy]** | 필수 | 초기 3가지 유형으로 제한 |
| `weight` | Float | 관계의 강도/중요도 점수 (선택적) | 0.85 | 선택 | 복잡도를 높일 때 사용 가능 |

### III. 개발 및 비즈니스 구현 계획 (Developer & Business Alignment)

**1. 기술 관점 (Developer):**
*   이 플로우는 **Graph DB의 트랜잭션 로직을 단 한 번만, 사용자 요청 시에 실행**하는 것에 초점을 맞춰야 합니다.
*   API 엔드포인트: `POST /api/generate-knowledge-graph`
*   핵심 로직: (1) 텍스트 파싱 $\rightarrow$ (2) NER(Named Entity Recognition) $\rightarrow$ (3) Relation Extraction 모델 호출 $\rightarrow$ (4) DB 트랜잭션 커밋.

**2. 수익화 관점 (Business):**
*   **가격 책정:** 이 플로우의 실행 자체가 **최초 크레딧 소비 지점**이 되어야 합니다. ("지식 구조 분석에 5 크레딧 소모")
*   **가치 포장:** 사용자에게는 "단순 추출" 결과물이 아닌, "**[X]가지 핵심 관계성 발견 및 자동 구조화 리포트**"라는 형태로 패키징하여 제공해야 합니다. 이는 사용자가 지불 가치를 느끼는 가장 중요한 요소입니다.
