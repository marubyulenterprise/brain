# 🧪 통합 테스트 스위트: `/api/predict-missing-edge`
*(Version: 1.0 - Atomic Transaction Focus)*

## 🎯 목표 API 엔드포인트 정의
*   **Endpoint:** `POST /api/predict-missing-edge`
*   **Purpose:** 사용자가 제공한 다중 데이터셋(콘텐츠, 트렌드, 시장 키워드 등)을 기반으로 지식 그래프 내의 'Missing Edge'를 예측하고 그 비즈니스 가치를 계산한다.
*   **핵심 원칙:** 모든 트랜잭션은 **원자적(Atomic)**이어야 한다. 즉, 성공적으로 Missing Edge가 발견되어 결과가 반환되거나 (Success), 아니면 입력 단계에서 오류가 감지되어 전체 프로세스가 롤백되어야 한다 (Failure). 중간 상태는 절대 노출되면 안 된다.

## ⚙️ 전제 조건 및 테스트 환경
1.  **Precondition:** 사용자는 유효한 `user_id`와 초기 크레딧(Credit)을 가지고 요청해야 함.
2.  **Input Schema:** 최소 3가지 이상의 데이터 타입을 포함해야 (예: Content Data, Trend Keyword Data, Market Segment Data).
3.  **Success Criteria:** Missing Edge가 발견되었음을 명확히 지적하고, 그에 따른 'Missing Edge Score'와 'Potential ROI ($)'를 계산하여 반환해야 함.

## 🔬 상세 통합 테스트 케이스 세트 (Test Case Set)

### I. 성공 시나리오 (Happy Path & Edge Discovery)
| TC ID | 목적 | 입력 데이터 조건 | 기대 결과값 (Expected Output) | 원자성 검증 포인트 |
| :---: | :--- | :--- | :--- | :--- |
| **TC-01** | **최적의 Missing Edge 발견 (Gold Standard)**. 가장 이상적인 시나리오로, 명확하고 높은 가치를 가진 Missing Edge를 예측하는 경우. | 3가지 이상의 데이터셋이 충분히 풍부하며, 서로 상충되지 않고 결합할 수 있는 구조가 존재함. | Status: `SUCCESS`. Missing Edge Score $\ge$ 임계값(Threshold). 상세한 Missing Edge 설명과 함께 ROI 값이 반환됨. | **Postcondition:** 크레딧 차감 완료 및 예측 결과 저장이 성공적으로 이루어져야 함. 롤백 필요 없음. |
| **TC-02** | **모호하나 잠재적 가치가 높은 경우**. 데이터 연결이 직접적이지 않으나, 분석 엔진이 여러 조합 중 하나의 Missing Edge를 제안해야 하는 경우. | 데이터셋 간의 관계성이 약하거나(Weak Correlation), 다수의 가능한 Edges가 존재함. | Status: `SUCCESS`. 가장 확률이 높거나 비즈니스 가중치가 높은 1순위 Missing Edge와 해당 신뢰도(Confidence Score) 반환. | **Postcondition:** 여러 가설 중 하나를 선택했으므로, 이 결정 과정의 트랜잭션 기록이 남아야 함. |

### II. 실패 및 경계 시나리오 (Failure & Boundary Cases - 원자성 테스트 핵심)
| TC ID | 목적 | 입력 데이터 조건 | 기대 결과값 (Expected Output) | 롤백/에러 처리 로직 |
| :---: | :--- | :--- | :--- | :--- |
| **TC-03** | **필수 데이터 누락 시 (Incomplete Data)**. 필수 Input 중 하나(예: Market Segment)가 아예 제공되지 않음. | 최소 3가지의 입력이 필요한데, 그중 한 가지가 `null` 또는 빈 배열로 전송됨. | Status: `FAILURE_INPUT`. 에러 메시지: "분석에 필수적인 [데이터 타입] 정보가 누락되었습니다." (구체적 가이드 제공). | **Rollback:** 어떠한 데이터도 읽거나 쓸 필요가 없으므로, 트랜잭션은 아예 시작되지 않아야 함. 크레딧 차감 없음. |
| **TC-04** | **데이터 상충/모순 발생 시 (Contradiction)**. 입력된 데이터 자체가 논리적으로 모순되어 그래프 생성이 불가능한 경우. | 예: "지역 A의 구매 의도"와 "지역 B의 트래픽 키워드"가 동시에 높은 중요도를 가지지만, 두 데이터를 연결하는 Edges가 존재하지 않는 구조적 오류. | Status: `FAILURE_LOGIC`. 에러 메시지: "제공된 데이터 간에 논리적 모순이 감지되었습니다. 입력 데이터 검토를 요청합니다." (충돌 지점 명시). | **Rollback:** 그래프 계산 엔진 실행 전 단계에서 실패해야 하며, DB에 임시 트랜잭션 상태가 남아서는 안 됨. 원본 Input은 변경되지 않음. |
| **TC-05** | **크레딧 부족 시 (Business Constraint)**. 분석을 시작하기 전에 크레딧이 충분하지 않은 경우. | `user_id`와 함께 요청했으나, 현재 잔여 크레딧이 API 호출에 필요한 비용(Cost)보다 적음. | Status: `FAILURE_CREDIT`. 에러 메시지: "Missing Edge 발견에 50 크레딧이 필요합니다. 잔액: X." (남은 금액 명시). | **Rollback:** 트랜잭션 시작 자체가 차단되어야 함. 데이터베이스 접근 전에 비즈니스 로직 계층에서 체크 완료. |

### III. 시스템 제약 및 성능 시나리오
| TC ID | 목적 | 입력 데이터 조건 | 기대 결과값 (Expected Output) | 롤백/에러 처리 로직 |
| :---: | :--- | :--- | :--- | :--- |
| **TC-06** | **처리 시간 초과 (Timeout)**. 그래프 복잡도가 매우 높거나, 데이터 양이 비정상적으로 많아 계산 시간이 시스템 제한(예: 15초)을 초과하는 경우. | 수백만 개의 노드와 관계성으로 구성된 초대형 가상의 지식 그래프를 분석하도록 요청. | Status: `FAILURE_TIMEOUT`. 에러 메시지: "분석이 지정된 시간 내에 완료되지 않았습니다. 데이터셋을 분할하여 재요청해 주십시오." (재시도 권장). | **Rollback:** 계산 엔진의 메모리 할당 및 상태 관리가 중요함. 실패 시, 내부 로그만 기록하고 사용자에게는 Clean Error를 반환해야 함. |
| **TC-07** | **데이터 무결성 검증 위반**. 입력 데이터 중 일부가 기대하는 스키마(Schema)와 다르게 잘못된 타입으로 전송되는 경우 (예: 날짜 필드에 문자열 전송). | `data_type`을 엄격히 검사해야 하는 필드에 유효하지 않은 데이터를 포함하여 전송. | Status: `FAILURE_SCHEMA`. 에러 메시지: "필드 [Field Name]은(는) 반드시 YYYY-MM-DD 형식의 날짜여야 합니다." (오류 위치 명시). | **Rollback:** 입력 파싱 단계에서 실패를 감지하고, DB 접근 전에 요청을 거부해야 함. 트랜잭션 시작 불가. |
