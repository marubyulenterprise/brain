# 📄 비즈니스 규칙 정의서 (BRD): $L$ Amount 기반 D-Sub Paywall 로직
**문서 버전:** v1.0
**작성일:** 2026년 5월 18일
**목표:** 사용자가 서비스 이용 한계에 도달했을 때, 단순한 '사용 불가' 메시지가 아닌, **정량화된 재무적 손실($L$ Amount)**을 인지하게 하여 구독(D-Sub) 결제를 유도하는 Paywall UX/UI의 핵심 로직 정의.

## 1. 개요 및 범위 (Scope & Goal)
*   **목표:** 사용자의 잠재적인 데이터 결핍($L$ Amount)을 극대화하여, 서비스 가치를 '필수 비용(Essential Cost)'으로 인식하게 한다.
*   **대상 기능:** 모든 크레딧 소모 기능을 호출하는 API 게이트웨이 레벨의 예외 처리 및 Paywall UI/UX 흐름.
*   **핵심 원칙:** 시스템은 사용자가 *무엇을 잃고 있는지*에 대한 공포(Fear of Loss)를 시각화해야 한다.

## 2. 재무적 조건 정의 (Financial Triggers & Mechanics)

| 단계 | 발생 조건 (Trigger) | $L$ Amount 계산 로직 | UI/UX 요구사항 (Shock Element) |
| :--- | :--- | :--- | :--- |
| **A. Free Tier** | 최초 가입, 크레딧 잔량 50% 이상. | N/A (초기 손실액 없음). | 친근하고 교육적 안내. "최대 $X$ 만큼의 지식 자산을 만들 수 있습니다." (가치 부여) |

## 3. 상세 비즈니스 규칙 (Business Logic Rules)

### 3.1. $L$ Amount 계산 공식 (Critical Failure 시점)
$$ L_{total} = \text{Max}(\text{Current Loss}, (\text{Source Volume} \times \text{Avg Value per Unit})) $$
*   **Source Volume:** 사용자가 시스템에 업로드한 원본 콘텐츠 데이터 총량 (MB 또는 페이지 수).
*   **Avg Value per Unit:** 우리 서비스가 이 데이터를 구조화하여 추출하는 평균적인 노동 가치. (이 값이 높을수록 $L$ Amount도 높아짐)

### 3.2. 타이밍 및 인터랙션 규칙
1.  **경고 임계값 도달 시점:** 사용자가 기능을 호출할 때마다 API 게이트웨이가 크레딧 소진율(%)를 계산하여, **실시간으로** 잔여 $L$ Amount가 증가하는 애니메이션을 배경에 지속적으로 표시해야 한다. (시각적 압박감 유지)
2.  **Paywall 진입:** 사용자가 기능을 호출할 때마다 `Credit Exhausted` 에러 코드를 반환하고, 이와 동시에 **모든 비활성화된 버튼과 UI 요소가 '비용 발생'을 암시하는 붉은색 필터(Red Overlay)**로 덮여야 한다.
3.  **재도전 유도:** Paywall 페이지 하단에는 결제 옵션 외에, "자주 사용하는 기능 목록"을 보여주되, 각 기능 옆에 **"이 기능을 사용하지 못하면 $Y$ 만큼의 손실액이 발생합니다."**라는 문구를 배치해야 한다.
